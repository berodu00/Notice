## 1. 프로젝트 개요

### 1.1 배경 및 목적
기존 이메일 기반의 IT 공지 업무는 이력 관리의 어려움과 비효율성을 내포하고 있습니다. 이에 따라 공지 업무 프로세스를 시스템화하고, MS Graph API를 활용하여 Outlook 일정과 자동 연동되는 통합 관리 시스템을 구축하고자 합니다.

### 1.2 핵심 목표
* **공지 업무 시스템화:** 웹 기반 공지 등록 및 결재 프로세스 구축.
* **Outlook 연동 자동화:** MS Graph API를 활용하여 공지 등록 시 Outlook 캘린더(자원 일정)에 자동 반영.
* **다채널 알림 통합:** 이메일뿐만 아니라 사내 메신저(GW), 포털, 모바일 앱(KZ이음)으로 알림 채널 확장.
* **사용자 편의성 증대:** 일반 임직원을 위한 로그인 불필요(SSO) 통합 뷰 제공 및 완료된 점검 일정의 시각적 상태 업데이트.

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 공지 등록 및 결재 (Admin)
* **공지 속성 관리:** 공지 유형(일반, 점검, 장애 등), 영향받는 서비스, 중요도, 수신 대상(법인/부서) 설정.
* **발송 예약:**
    * **정기 발송:** 오전 08:30, 오후 17:30에 모아서 일괄 발송하는 옵션 제공.
    * **즉시 발송:** 긴급 상황 시 승인 즉시 발송.
* **결재 프로세스:** 작성 후 담당 매니저 승인을 거쳐야 실제 발송 및 일정 등록 수행 (반려 기능 포함).

### 3.2 Outlook 연동 (Core Logic)
* **자원(Resource) 활용:** 개인 캘린더가 아닌 별도의 'IT 공지일정' 자원(회의실과 유사한 개념)을 생성하여 일정을 등록함.
* **자동 동기화:** 시스템에서 승인된 공지는 MS Graph API를 통해 해당 자원의 캘린더에 이벤트를 생성하고, 수신자에게 메일을 발송함.

### 3.3 완료 공지 처리 (State Management)
* **상태 업데이트:** 점검이나 장애 조치가 완료된 경우, 별도의 '완료 공지' 메일을 재발송하는 대신, 기존 캘린더 일정의 색상이나 상태를 '완료(회색 등)'로 변경하여 시각적으로 인지시킴.
* **연계성:** 완료 공지 작성 시 기존 진행 중인 공지를 선택하여 매핑하는 기능 제공.

### 3.4 알림 채널 확장
* **Multi-Channel Support:** 이메일 외 KZ이음(모바일), 그룹사 포탈, GW 메신저로 알림을 전송할 수 있는 표준 API 연계 가이드 수립 및 구현.

## 4. UI/UX 요구사항

### 4.1 관리자 (Back Office)
* **Dashboard:** 결재 대기, 발송 예정/완료/실패 건수를 카드 형태 UI로 직관적으로 표시. 공지 유형별/부서별 통계 차트 제공.
* **등록 화면:** 자주 사용하는 수신처나 서비스 목록을 템플릿화하여 선택 편의성 제공. 정기/즉시 발송 선택 라디오 버튼 배치.
* **History:** 전체 발송 이력을 리스트로 제공하며, 승인 상태(대기/완료/반려)를 배지(Badge) 형태로 구분하여 가시성 확보.

### 4.2 일반 사용자 (Front Office)
* **접근성:** 별도의 시스템 로그인 없이 SSO(Single Sign-On)를 통해 접근 가능한 'IT 공지 현황판' 페이지 제공.
* **Timeline View:** 월간/주간/일간 캘린더 뷰를 제공하며, 현재 진행 중인 작업과 완료된 작업을 색상으로 구분 (예: 진행-파랑, 긴급-빨강, 완료-회색).
* **Mobile Web:** 모바일 앱(KZ이음) 내 웹뷰에서도 깨지지 않는 반응형(Responsive) 디자인 적용 필수.

## 5. 데이터베이스 설계 전략 (Draft Schema)

핵심 기능을 지원하기 위한 PostgreSQL 테이블 설계 방향입니다.

```sql
-- 1. 공지 마스터 테이블
CREATE TABLE notice_master (
    notice_id SERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT,
    notice_type VARCHAR(50), -- 일반, 점검, 장애 등
    severity_level VARCHAR(20), -- 상, 중, 하
    target_corp VARCHAR(100), -- 수신 법인
    target_dept VARCHAR(100), -- 수신 부서
    creator_id VARCHAR(50),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status VARCHAR(20) -- DRAFT, PENDING, APPROVED, REJECTED, COMPLETED
);

-- 2. 일정 상세 테이블 (Outlook 연동용)
CREATE TABLE notice_schedule (
    schedule_id SERIAL PRIMARY KEY,
    notice_id INTEGER REFERENCES notice_master(notice_id),
    start_time TIMESTAMP NOT NULL,
    end_time TIMESTAMP NOT NULL,
    is_regular_send BOOLEAN DEFAULT FALSE, -- 정기 발송 여부
    outlook_event_id VARCHAR(255) -- MS Graph API 연동 ID
);

-- 3. 결재 이력 테이블
CREATE TABLE approval_log (
    log_id SERIAL PRIMARY KEY,
    notice_id INTEGER REFERENCES notice_master(notice_id),
    approver_id VARCHAR(50),
    action VARCHAR(20), -- APPROVE, REJECT
    comment TEXT,
    action_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
